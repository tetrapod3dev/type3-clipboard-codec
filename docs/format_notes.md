# Format Notes

이 문서는 TYPE3 Edit의 클립보드 바이너리 포맷에 대해 현재까지 관찰된 구조와 해석 가설을 정리한 작업 문서입니다.

## 문서 목적

- 관찰된 바이너리 구조를 체계적으로 기록
- 객체 타입별 식별 패턴과 필드 의미를 정리
- 확정된 정보와 가설을 구분
- 파서 구현 시 참고할 기준 문서 역할 수행

## 상태 구분 규칙

각 항목은 다음 상태 중 하나로 관리합니다.

- **Confirmed**: 샘플 비교와 동작 검증으로 의미가 상당히 확정된 항목
- **Likely**: 정황상 매우 유력하지만 추가 샘플 검증이 필요한 항목
- **Unknown**: 아직 의미를 확정하지 못한 항목
- **Reserved Raw**: 의미 미확정 상태로 원시 값 보존이 필요한 항목

---

## 1. 전체 포맷 개요

현재까지의 관찰 기준으로 TYPE3 clipboard payload는 다음 요소들로 구성되는 것으로 추정됩니다.

1. 헤더 또는 고정 메타데이터 영역
2. 객체 레코드가 순차적으로 나열되는 본문 영역
3. 일부 객체별 가변 길이 데이터 영역
4. 문자열 또는 추가 속성 블록으로 보이는 후행 데이터

정확한 전체 스펙은 아직 확정되지 않았으며, 객체 종류에 따라 중간 구조가 달라질 수 있습니다.

---

## 2. 공통 관찰 사항

### 2.1 엔디언
- **Likely**: 다수 필드가 little-endian 정수/실수 구조를 따르는 것으로 보임

### 2.2 좌표 데이터
- **Likely**: 객체 좌표 관련 필드는 일정한 수치 패턴을 가지며, 샘플 간 비교 시 위치값 변화와 상관이 있음
- **Unknown**: 실제 단위 체계, 스케일 팩터, 좌표 원점 기준은 추가 검증 필요

### 2.3 객체 경계
- **Likely**: 특정 타입 식별자 또는 레코드 선두 패턴에 의해 객체 시작 지점을 구분할 수 있음
- **Unknown**: 모든 객체 타입이 동일한 레코드 헤더 구조를 공유하는지는 미확정

### 2.4 문자열
- **Likely**: 텍스트 객체에는 문자열 길이 또는 문자열 블록에 해당하는 가변 데이터가 포함됨
- **Unknown**: 인코딩 방식, 문자열 종료 방식, 스타일 정보 배치 방식은 추가 확인 필요

---

## 3. 객체 타입별 메모

## 3.1 Rectangle

### 관찰
- 사각형 계열 객체는 비교적 안정적인 좌표 필드 집합을 가지는 것으로 보임
- 시작점과 끝점 또는 bounding box 형태의 구조를 가질 가능성이 높음

### 추정 필드
- **Likely**: x1
- **Likely**: y1
- **Likely**: x2
- **Likely**: y2

### 미확정 항목
- 선 두께
- 채움 여부
- 레이어/속성 플래그
- 회전 관련 필드 존재 여부

---

## 3.2 Line

### 관찰
- 선 객체는 두 점 좌표를 가지는 단순 구조일 가능성이 높음
- 일부 샘플에서 사각형보다 짧은 구조를 가질 수 있음

### 추정 필드
- **Likely**: start_x
- **Likely**: start_y
- **Likely**: end_x
- **Likely**: end_y

### 미확정 항목
- 선 스타일
- 화살표 속성
- 폭 또는 펜 설정

---

## 3.3 Arc

### 관찰
- Arc는 단순 선/사각형보다 구조가 복잡하며, 제어점 또는 중심/반지름 기반 데이터가 포함될 가능성이 있음
- 샘플 분석 중 circular arc 관련 데이터에서 기대한 수의 점이 모두 드러나지 않는 사례가 있었음

### 현재 가설
- **Likely**: 시작점, 끝점 외에 추가 제어 정보가 별도 필드 또는 압축된 형식으로 저장될 수 있음
- **Unknown**: 제어점 1개 구조인지, 중심점+반지름+각도 구조인지, 또는 내부 플래그에 따라 표현 방식이 달라지는지 확인 필요

### 미해결 이슈
- circular arc에서 기대한 3개 점(시작점, 제어점, 끝점)이 항상 노출되지 않는 문제
- 일부 좌표가 공통 헤더나 별도 블록에 저장될 가능성
- 객체 종류별로 arc 표현 방식이 다를 가능성

---

## 3.4 Circle

### 관찰
- 중심점 + 반지름 또는 bounding box 기반 표현 중 하나일 가능성이 있음
- 샘플 비교 시 직관적인 원 객체 패턴이 존재하는 것으로 보임

### 현재 가설
- **Likely**: 중심 좌표 관련 필드 존재
- **Likely**: 반지름 또는 지름에 대응하는 수치 필드 존재
- **Unknown**: ellipse와 circle이 동일 구조인지 별도 타입인지 불명확

---

## 3.5 Rounded Rectangle

### 관찰
- 사각형 구조에 추가적인 corner radius 성격의 필드가 포함되는 것으로 추정
- Rectangle과 상당 부분 공통 구조를 공유할 가능성이 높음

### 현재 가설
- **Likely**: 기본 bounding box 필드 존재
- **Likely**: radius 또는 rounding 관련 추가 필드 존재

### 미확정 항목
- x/y 방향 radius 분리 여부
- 내부 corner style 플래그 존재 여부

---

## 3.6 Text

### 관찰
- 문자열 데이터와 위치/정렬/스타일 계열 정보가 함께 포함될 가능성이 높음
- 다른 도형 객체보다 가변 길이 데이터가 포함될 가능성이 큼

### 현재 가설
- **Likely**: 텍스트 본문
- **Likely**: 삽입 위치 또는 기준점 좌표
- **Unknown**: 폰트 정보, 높이, 정렬, 문자 간격, 회전 관련 필드 배치

### 미확정 항목
- 문자열 인코딩 방식
- 문자열 길이 필드 형식
- 멀티라인 처리 방식
- 스타일 블록과 본문 문자열의 경계

---

## 4. 파서 설계 메모

### 설계 원칙
- 확정된 필드만 강하게 모델링
- 미확정 필드는 raw bytes 또는 raw value로 보존
- 객체 타입 감지와 객체 본문 파싱을 분리
- 필드 의미가 불명확한 경우 이름을 섣불리 확정하지 않음

### 권장 표현 방식
예:
- `unknown_01`
- `reserved_02`
- `candidate_radius`
- `raw_tail_bytes`

이런 식으로 “모른다”는 사실 자체를 구조에 반영하는 편이 역공학 프로젝트에서는 유리합니다.

---

## 5. 샘플 분석 시 체크리스트

새 샘플을 분석할 때는 아래를 함께 기록합니다.

- 객체 종류
- 복사한 객체 수
- 좌표/크기/텍스트 내용을 어떻게 바꿨는지
- 이전 샘플과 달라진 바이트 범위
- 반복적으로 고정되는 바이트 범위
- 길이 필드로 의심되는 값
- 문자열/좌표/플래그 후보 필드

---

## 6. 미해결 질문

- 객체 레코드 시작을 완전히 신뢰할 수 있는 시그니처가 있는가
- 모든 객체가 공통 헤더를 가지는가
- 좌표값은 정수인가, 고정소수점인가, 부동소수점인가
- Arc는 어떤 수학적 표현으로 저장되는가
- Text의 문자열 인코딩은 무엇인가
- 스타일 정보와 지오메트리 정보는 어떤 순서로 배치되는가
- 객체 간 공통 속성 블록이 존재하는가

---

## 7. 문서 운영 원칙

이 문서는 확정 문서가 아니라 작업 문서입니다.  
새 샘플 분석 결과가 나오면 아래 원칙으로 갱신합니다.

1. 확정 정보와 가설을 반드시 구분한다.
2. 기존 가설이 틀린 경우 흔적을 남기고 수정한다.
3. 파서 구현보다 문서 정합성을 우선한다.
4. “모르는 상태”를 숨기지 않는다.
